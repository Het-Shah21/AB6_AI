# Task 1.2 — Project Files: `.gitignore`, `.env.example`

## System Design Reference

Standard project hygiene per the Master System Design's "Repository Setup" section. Every Python project needs to exclude build artifacts, virtual environments, IDE config, and secrets from version control.

---

## File: `.gitignore`

```
# Python
__pycache__/
*.py[cod]
*.egg-info/
dist/
build/
```

**Line by line:**

- `__pycache__/` — Compiled Python bytecode (`.pyc` files). Generated automatically on import. Should never be committed because they're machine-specific and change with every Python version.
- `*.py[cod]` — Wildcard for `.pyc` (compiled), `.pyo` (optimized), `.pyd` (Windows DLL). Same reasoning as above.
- `*.egg-info/` — Package metadata directory created during `pip install -e .`. Contains version info, dependency list, entry points. Regenerated from `pyproject.toml` on each install.
- `dist/` — Distribution archives (`.tar.gz`, `.whl`) created by `python -m build`. Generated artifacts, not source.
- `build/` — Build staging directory used by setuptools. Intermediate files only.

```
# Virtual environment
venv/
.venv/
```

Shell-specific: PowerShell uses `venv/` as the convention for `python -m venv venv`. Never commit the venv — it's full of platform-specific binaries and pip-installed packages.

```
# IDE
.vscode/
.idea/
*.swp
*.swo
```

VS Code, PyCharm, and Vim swap files. These are personal editor settings; committing them creates conflicts between developers with different preferences.

```
# Environment
.env
```

**Critical for security.** The `.env` file contains secrets (database passwords, API keys). The `.env.example` is committed instead (see below).

```
# OS
.DS_Store
Thumbs.db
```

macOS Finder metadata (`.DS_Store`) and Windows thumbnail cache (`Thumbs.db`). OS-specific junk files.

```
# Testing
.htmlcov/
.coverage
.pytest_cache/
```

`htmlcov/` — HTML coverage report from `pytest --cov --cov-report=html`. `.coverage` — SQLite database of coverage data. `.pytest_cache/` — pytest's internal cache of previous test results (used for faster re-runs). All generated by running tests; not source code.

```
# Alembic
alembic/versions/*.pyc
```

If Alembic auto-generates compiled bytecode for migration scripts, ignore it.

```
# Logs
*.log
```

Log files generated at runtime (`uvicorn.log`, `alembic.log`, etc.). Never commit logs — they contain runtime data and can grow huge.

## File: `.env.example`

```
# Database
DATABASE_URL=postgresql+asyncpg://ab6:ab6_pass@localhost:5432/ab6_ai
```

Template for the production connection string. Shows the format `postgresql+asyncpg://user:password@host:port/database`. The `+asyncpg` tells SQLAlchemy to use the asyncpg driver. The `.env` file (not `.env.example`) would have the actual password.

```
# Redis
REDIS_URL=redis://localhost:6379/0
```

Redis connection string. `redis://` is the protocol, `localhost:6379` is default host:port, `/0` is Redis database number 0 (Redis has 16 logical databases, 0-15).

```
# LLM API Keys
OPENAI_API_KEY=
ANTHROPIC_API_KEY=
GOOGLE_API_KEY=
```

Empty in the example. Users fill in their own keys. The `settings.py` reads these and passes them to LangChain's `init_chat_model()`. Without them, the LLM calls fail and the fallback chain kicks in (ultimately using hardcoded demo text).

## How These Connect

- `.gitignore` protects the repo from accidentally committing `.env` (which has passwords) and `__pycache__/` (which is machine-specific).
- `.env.example` is the **contract** between the code and the developer: it defines every env var the application expects. The actual `.env` file is created by copying `.env.example` and filling in real values.
- `src/config/settings.py` reads env vars at import time via `pydantic-settings`. The `model_config = {"env_file": ".env"}` directive tells Pydantic to load from the `.env` file.
